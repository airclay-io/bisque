# API Surface

This page maps the public API by import path. Method signatures, trait impls,
fields, and detailed behavior are generated by rustdoc. Keep this page aligned
with the crate-level rustdoc in `src/lib.rs`, the domain module docs, and
`tests/api.rs`.

Public items are grouped by role. Processor implementations live under
feature-gated domain modules; host and authoring contracts have dedicated
modules. The crate root exposes the `params!` macro but does not re-export the
contract types.

## Feature Gates

| Feature | Default | Public module |
| --- | --- | --- |
| `filters` | Yes | `bisque::filters` |
| `dynamics` | Yes | `bisque::dynamics` |
| `mastering` | Yes | `bisque::mastering` |
| `analysis` | Yes | `bisque::analysis` |
| `generators` | Yes | `bisque::generators` |
| `time` | Yes | `bisque::time` |
| `repair` | Yes | `bisque::repair` |
| `spectral` | No | `bisque::spectral` |
| `test-support` | No | `bisque::testing` |
| `snapshot-support` | No | hidden repository snapshot cases |

`snapshot-support` enables repository tooling built on `test-support`. Its
snapshot-case module is public only so integration tests and `xtask` can share
generated cases. It is hidden from rustdoc and is not supported downstream API.

### `bisque::host`

`PreparedProcessor<P, T = f32>` owns one successfully prepared processor, its
`ProcessSpec`, and the next processing `sample_pos`. It provides in-place,
output-only, and split conveniences, restart/seek, checked immediate writes,
readouts, and flush delegation. It owns no processor collection, graph, file,
transport, event queue, plugin lifecycle, or cumulative tail policy.

## `bisque::dsp`

Use `bisque::dsp` for lower-level DSP utilities and wrappers.

| Path | Public items |
| --- | --- |
| `bisque::dsp` | `SmootherBank`, `db_to_linear`, `linear_to_db_floor` |
| `bisque::dsp::math` | `sin`, `cos`, `tan`, `sqrt`, `hypot`, `atan2`, `exp`, `ln`, `log10`, `pow` |
| `bisque::dsp::oversample` | `PolyphaseUpsampler` |

The raw block driver is internal. Hosts construct
`processor::KernelProcessor<K, T = f32>` directly or through
`Kernel::into_processor()`.

## Settings Structs

Public settings structs are `#[non_exhaustive]` so new fields can be added
without breaking downstream code. Construct them with `Default`, `new()`, and the
chainable setter methods, not struct literals.

## `bisque::time`

Feature: `time`

| Public item | Role |
| --- | --- |
| `Delay` | Integer-sample feedback delay |
| `DelaySettings` | Named construction settings for `Delay` |
| `DelayParams` | Smoothed parameter values for `Delay` |
| `TimeStretch` | Overlap-add variable-rate time stretching |
| `TimeStretchSettings` | Named construction settings for `TimeStretch` |

Parameter constants:

| Type | Constants |
| --- | --- |
| `Delay` | `DELAY_MS`, `FEEDBACK`, `MIX` |

`Delay` rounds delay time to whole samples and uses a clean feedback path
without damping filters or saturation. Delay-time automation crossfades
between integer taps over one control-rate cell, so it is click-safe.

`TimeStretch` has no automatable parameters. It is plain
overlap-add, not a phase vocoder. It supports 1 to 16 channels. Its requested
ratio is available through `stretch()`, and its rounded effective ratio is
available through `effective_stretch()` after preparation.

## Maintenance

When public API changes, update these files together.

- `src/lib.rs` crate-level rustdoc API surface
- the relevant `src/<domain>/mod.rs` public API list
- this page
- `tests/api.rs`

For method-level changes on existing types, rustdoc is authoritative and this
page usually does not need to change.
